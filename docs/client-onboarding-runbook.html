Avoqado Sync Service · Runbook de Onboarding

Procedimiento canónico para conectar un cliente SoftRestaurant nuevo a la sincronización Avoqado en tiempo real.
Regla de oro del orden: el venue debe existir primero en el backend de Avoqado. Si el servicio empieza a publicar eventos de un venue que no existe en el backend, esos eventos se pierden (dead-letter) y los heartbeats disparan configuration.error. Backend → DB → servicio.

0Compatibilidad (v10 / v11 / v12 — mismo binario)

Motor: SQL Server 2014 Express (32-bit) para todas las versiones (v12 es la versión del POS, no del motor). El formato de entity-id se decide por presencia de la columna WorkspaceId, no por el número de versión:

  • v11 / v12 y cualquier v10 con WorkspaceId → path WorkspaceId. ✅ validado en vivo.
  • v10 puro (sin WorkspaceId) → {Instance}:{IdTurno}:{Folio}. (Raro en la práctica — validar contra un v10 limpio real antes de go-live.)

1Prerrequisitos

En el backend de Avoqado (PRIMERO)

  • Crear el Venue en la plataforma → anotar su venueId.
  • Poner posType = SOFTRESTAURANT al venue (requerido para recibir comandos).
  • Confirmar que el backend está conectado al broker RabbitMQ compartido (el mismo que usará el servicio).

En el servidor del POS del cliente

  • Node.js 18 (o usar el AvoqadoSyncService.exe empaquetado, que incluye Node).
  • Permisos de Administrador.
  • Salida de red al broker (puerto 5671 TLS / 5672).
  • SQL local accesible: localhost\NATIONALSOFT.

2Paso a paso

1 · Backup de la base de datos

BACKUP DATABASE [DB_NAME] TO DISK = 'C:\Backups\DB_NAME_PRE_AVOQADO.bak' WITH FORMAT, INIT, COMPRESSION

2 · Detectar versión (sanity check)

SELECT versiondb FROM parametros2;                        -- 10.x / 11.x / 12.x
SELECT COL_LENGTH('tempcheques','WorkspaceId') AS HasWID;  -- 16 = path WorkspaceId, NULL = v10 puro

3 · Instalar los objetos SQL de Avoqado

Contra la base del venue. El script fija QUOTED_IDENTIFIER ON, así que -I es opcional:

sqlcmd -S "localhost\NATIONALSOFT" -U sa -P <PW> -d <DB_NAME> -i "scripts\sql\01-COMPLETE-INSTALL.sql"

Si hubo una instalación previa/parcial, correr antes 00-CLEANUP-ALL.sql.

4 · Verificar la instalación

sqlcmd -S "localhost\NATIONALSOFT" -U sa -P <PW> -d <DB_NAME> -i "scripts\sql\00-VERIFICATION.sql"

Esperado: 7 tablas Avoqado*, los stored procedures, 4 triggers HABILITADOS, métodos de pago ACASH/ACARD.

5 · Configurar el servicio (producción)

Producción lee %ProgramData%\AvoqadoSync\config.json (NO .env; NODE_ENV no debe ser development):

{
  "venueId": "<VENUE_ID_DEL_BACKEND>",
  "posType": "softrestaurant",
  "posVersion": "11",
  "rabbitMqUrl": "amqps://user:pass@<BROKER_COMPARTIDO>/vhost",
  "sqlConfig": {
    "server": "localhost",
    "instanceName": "NATIONALSOFT",
    "database": "<DB_NAME>",
    "user": "sa",
    "password": "<PW>",
    "options": { "instanceName": "NATIONALSOFT", "encrypt": false, "trustServerCertificate": true }
  }
}
⚠️ El rabbitMqUrl debe ser el MISMO broker que consume el backend — si no, los eventos nunca llegan.

6 · Instalar y arrancar el servicio (como Administrador)

npm run svc:install            # registra "Avoqado POS Sync Service" y lo arranca (auto-start al boot)
Get-Service -Name "Avoqado POS Sync Service"

3Checklist de go-live

  • Logs (logs\info-YYYY-MM-DD.log): Versión detectada · SQL Server establecida · RabbitMQ establecida · Vinculando consumer(s).
  • Heartbeat cada 60s: ❤️ Latido enviado.
  • Backend: el venue muestra PosConnectionStatus = ONLINE / posStatus = CONNECTED.
  • Crear una orden real en el POS → aparece en el backend (fila Order / dashboard).
  • Cerrar turno en el POS → sin eventos "deleted" espurios; se publica shift.closed.

4No interferencia con el POS

Los triggers son set-based (multi-fila), con SET NOCOUNT ON y TRY/CATCH que registra pero NO hace ROLLBACK → un fallo de tracking nunca aborta una orden/pago del POS.
  • El polling lee la tabla aparte AvoqadoTracking (indexada), cada 2s, con guarda anti-solape.
  • Siempre correr 00-VERIFICATION.sql tras instalar — el único riesgo residual es una tabla de tracking ausente/rota, que la verificación detecta.
  • El Comandante (Avoqado → POS, ej. Payment.APPLY) sí escribe en tablas vivas por diseño; es transaccional y corto.
